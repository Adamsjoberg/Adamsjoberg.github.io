TagAlong

During this project I worked with a team where our task was to create an application for Wikimedia. The issue at hand concerned uncategorized images, meaning that when users searched for images through specific categories, the images were not found

Info

This project took place at the department of Informatics, Umeå University, Autumn 2014.

Background

Wikipedia's media database Wikimedia Commons contains more than 18 million images that are free to use for everyone. One way to make the images easier to find and filter, is to add images into categories. There is a categorization system on Wikimedia Commons today, but the problem is that many people, especially beginners, do not take the time needed to add pictures to a category. It is not always easy to figure out to which of the available categories an image belongs. In the current situation, there are about 400,000 uncategorized pictures and some 1 000 new uncategorized images are added every day.

Task

Develop and present a design proposal and an interactive prototype.

Role

Conceptual design, UX designer, Interaction designer, Visual designer and mediator in User evaluations.

Approach

As a team we had a closer look on the Wikimedia website and were trying to understand the problem in depth. It was very obvious to us that Wikimedia was using categorization wrong and actually meant tagging. This easily led to confusion when uploading an image. Our focus therefore was to tag images and not to put them in categories.

Individually we did research about gamification. Afterwards we started discussing back and forth what good gamification is and what it means. To get going and getting inspired, we had a look at successful mobile games as well as tagging games. All the tagging games had one thing in common, they felt like work.

Navigation and Main page

One of the first major changes we decided upon was to recede the size of the main menu. It is easy to make it smaller, using an accordion, so that menus can be expanded and collapsed. One of our research findings was, that knowing about specific topic areas which needed help would turn non-contributors into participants. We decided that this was an important feature to make people both aware of what they can do, but also encouraging them to do so by making it more accessible. So we added “Take part!” to the menu, with prominent green background, an overview about different possibilities of taking part. We also changed the “Interaction” menu to “Community” instead, as that is what you’re interacting with - the community.

To deal with the hidden search syntax issue on Wikipedia, we added a feature that already exists on the Wikimedia Commons site. We added, beneath the search function, options to search in specific parts of the site. By adding this search options and moving the search next to the Wikipedia logo, we increase availability, by increasing awareness and accessibility.

Another issue was the cluttered main page, where all kinds of featured texts crowd one another. Even though all these features are something we want to make more prominent, we also want to make them more attractive and less intimidating. We chose to design them similarly to common widgets on other sites.

The green color of “Take part!” can also been found in the Article page. As underline in the tab navigation as well as the edit link next to the Article subheadings. Our goal is to make the user subconsciously link these together, the user sees a green color and subconsciously feels inclined to take part.

Edit page

Although the Visual Editor is activated by default in for example Swedish Wikipedia and in German language variations, it is not activated in the main English and German Wikipedia. In the research, we found that a fair amount of people not contributing to Wikipedia stated that not knowing the syntax of the markup language hinders them to contribute.

Additionally, our own experiences with the visual editor led us to the suggestion that activating the visual editor by default would be an important step in attracting a more diverse editor group. People that make the first steps in Wikipedia on their own might not make it that far to activate the visual editor in the settings section so they might not become editors. Especially if they are not familiar with markup language and are unwilling to make an effort to learn it.

Although this should be known to the core Wikipedians and even though core editors would be able to turn the visual editor off, they still blocked the visual editor in 2013 for various reasons. Therefore we suggest to the foundation to make another try to convince their users to activate the visual editor for at least for new users.

Debate page

We changed the debate page to make it look like a discussion forum. The threads are automatically generated based on the headings in the article itself. On the right we display widgets with the debate rules and to the article related projects. The rules widget keeps a yellow color as in the current Wikipedia talk page. The projects widget has the green color, again we want to imply that green means to take part and these projects are taking part.

History page

The history page in the current Wikipedia is flooded with unnecessary and partially confusing information. Our approach has a revision column next to the article. Like Google does it with the Google docs revision history. We display the change date, username and the comment. When a user selects the revision, the change gets displayed in the article with green color. This makes it easier to comprehend changes that have been made.

User page

Even though we didn’t want to implement that many more new social features, we still wanted to make user pages more prominent. In contrast to other social networks the user page should not be mostly about the users’ identity but about helping them to connect to people to similar interests.

As it stands, user pages look like articles. We still want people to be able to have that kind of user page, but we wanted to give users a base template to start their user page from. If they want to deviate from the base template, they are free to do so, but by providing a base template we put certain expectations on the user. By making these expectations more of a norm, over time, there will be more information on the average user page, increasing social aspects in a not too intrusive manner.

In the proposed template is a subcategory named ‘Fields of Focus’. This is supposed to help users to connect to each other. Fields of Focus represent specific subjects that the user is engaged in. By connecting the fields to the users account we make it more accessible for users to find people who are interested in the same topics. This feature is not meant to replace the already existing “WikiProjects” but rather as a way to facilitate the way users connect to each other, hence enabling an easier way of finding potential partners to contribute with.

Summary

A serious shortcoming of our proposal is that core-Wikipedians will most certainly be reluctant to implement our suggestions. As we have seen with the visual editor it is very hard for the Wikimedia foundation to propose changes to their user base. We know therefore that it is quite unlikely that they will be successful in persuading their users of a redesign of the platform. However, we still decided to at least create an initial redesign concept to give a clearer picture of what we mean by improving the accessibility of the site. We believe that by lowering the technical barriers Wikipedia would be successful in attracting a more diverse user group and that would benefit the society because there would be a much broader knowledge for everyone available on Wikipedia.
